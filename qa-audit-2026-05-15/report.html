P1 High

P1
Dossier iframe sandbox escape (stored-XSS attack surface)
/leads/$leadId · category: security
P1-01

Evidence

Browser console emits: "An iframe which has both allow-scripts and allow-same-origin for its sandbox attribute can escape its sandboxing."

The dossier iframe is rendered as <iframe srcdoc={lead.html} sandbox="allow-scripts allow-same-origin">. lead.html is sourced verbatim from the user-uploaded HTML on /upload (verified via GET /server/api/leads/<id>). Because the iframe shares the parent origin and can execute scripts, any uploaded HTML containing <script> can read cookies/localStorage and call /server/api/* with the victim's credentials.

Reproduction

  1. Log in as any user. Navigate to /leads/31210000000141422.
  2. Open DevTools → Console. Observe the sandbox-escape warning at page load.
  3. Inspect the dossier <iframe>. Confirm sandbox="allow-scripts allow-same-origin" and srcdoc populated.

Root cause

LeadDetailPage renders the dossier via srcdoc with both sandbox flags. functions/api/lib/parser.js only extracts structured fields — the full HTML body is stored verbatim in the Data Store leads.html column and in Stratus, and returned to the client unmodified.

Suggested fix

Either:

  • (a) Drop allow-same-origin from the sandbox and serve the dossier via a signed Stratus URL on a different host (sandbox escape requires both flags together).
  • (b) Run the uploaded HTML through DOMPurify with a strict allow-list before storing; keep srcdoc rendering but remove allow-same-origin.

Option (a) is the cleanest defense-in-depth: even if sanitizer is bypassed, scripts can't reach the parent origin.

Unresolved template placeholder <rep name> rendered to users in email bodies
/leads/$leadId · category: feature-logic
P1-02

Evidence

Inside the dossier iframe email-templates section on lead 31210000000141422 (Marco Lisboa @ AIG), the literal text <rep name> appears in 3 email body positions. End users see the placeholder string unmodified.

Reproduction

  1. Log in. Navigate to /leads/31210000000141422.
  2. Scroll the dossier iframe to the outreach / email-templates section.
  3. Observe <rep name> verbatim where the rep's name should be inserted.

Root cause

The upstream dossier generator leaves the <rep name> token unsubstituted when the rep identity is unknown. The Hub renders the dossier verbatim with no token-replacement layer.

Suggested fix

Inject a render-time substitution in LeadDetailPage: replace <rep name> with the signed-in user's full name (already available from useAuth().user). Fall back to a neutral phrase like "your account team" if name is unavailable. Add a lint that warns on any <\w+ name> pattern left in dossier output.

Evidence

At 390×844 mobile viewport, 4 unlabeled <button> elements (40×32 px each) are present — one per user row. Each has no text content, no aria-label, no title. Each is the per-row Delete user destructive action.

P2 Medium

P2
ICP Ladder uses exact-match semantics despite "≥ N stars" UI copy
/ · /leads filter · feature-logic
P2-01

Evidence

Dashboard ICP Ladder reads "Click a row to drill into leads with at least that many stars." All 5 production leads return icp_rating: "Strong". Stats: icp_min=4 → 5 leads; icp_min=3/2/1 → 0 leads. Clicking ≥1★ shows 0 leads — direct violation of the "≥" contract. Adding the QA stub (icp_rating "Weak") makes icp_min=2 return 1 — confirming the bucket is exact-match, not threshold.

Reproduction

  1. On dashboard, click ICP Ladder row "≥ 1★" → /leads?icp_min=1 shows 0 leads.
  2. Click "≥ 4★" → shows the 5-lead "Strong" cohort.
  3. The empty-state UI is well-designed; the underlying count is the bug.

Root cause

Server /server/api/stats maps icp_rating string to a single bucket integer and the /leads filter compares with =, not >=.

Suggested fix

At ingestion, map ICP rating strings ("Weak"→2, "Moderate"→3, "Strong"→4, "Perfect"→5) to a numeric icp_stars column on the leads table. Then /server/api/leads?icp_min=N can do WHERE icp_stars >= N and dashboard stats can count via the same semantic.

P2
Signal labels case-sensitive: "Compliance Need" and "Compliance need" counted separately
/ Top Score Drivers · data-quality
P2-02

Evidence

Dashboard widget displays both "Compliance Need: 3" and "Compliance need: 2" as distinct top drivers. /server/api/stats returns both forms in the signals array. Visually presents as a UI duplicate.

Root cause

Dossier authors capitalize labels differently. parser.js stores attribution labels verbatim; the stats aggregator groups by exact string.

Suggested fix

Canonicalize at ingest (lower-case for grouping, title-case for display). Alternatively add a fuzzy-merge step in the aggregator that collapses entries whose lower-cased forms match.

P2
Parser accepts dimension scores exceeding their max
/upload pipeline · feature-logic
P2-04

Evidence

Uploaded a stub dossier with budget_score=22, budget_max=20. The API persisted both values verbatim. Composite score 42 = 5+5+10+22. Dashboard Budget H/M/L counters operate on the bad ratio.

Reproduction

  1. Upload a dossier with a <div class="dim-row"> where dim-score > dim-max.
  2. GET /server/api/leads/<new-id> → values stored verbatim.

Suggested fix

In functions/api/lib/parser.js, after extracting each dim, score = Math.min(score, max) and log a warning. Surface to admins as lead.parse_warnings JSON.

P2
Mobile: 67/100 interactive elements below WCAG 2.5.5 44×44 minimum
/, /leads (mobile) · accessibility
P2-06

Evidence

At 390×844 viewport, an in-browser script measured: hamburger 36×36; Upload/Browse-leads CTAs 32 px tall; "View all hot" 16 px tall; verdict-confidence band rows 12 px tall. 67 of 100 measured controls fail the 44 px minimum on one dimension.

Suggested fix

Add @media (pointer: coarse) rules bumping min-height and padding on all a/button within the SPA to ≥44 px. Alternatively a Tailwind plugin applying min-h-11 min-w-11 on a .touch-target base class.

P2
No edit / re-score / export / share controls on lead detail — only Delete
/leads/$leadId · feature-gap

Lead detail page exposes exactly: Sign out (nav), Back to leads, Delete, and three Copy-email buttons. Bad-data leads (e.g. P2-04 budget>max) can't be corrected without deleting and re-uploading the dossier.

No semantic <h1> for lead name on detail page
/leads/$leadId · accessibility
P2-10

Evidence

Outer SPA chrome on the detail page exposes no heading-role element for the lead's name — only StaticText nodes. document.querySelectorAll('h1').length in the outer document returns 0 (h1s exist only inside the embedded dossier iframe).

Wrap the lead-name display element in <h1> (or visually-hidden screen-reader-only h1 if the design must stay flat). Demote internal iframe heading to h2 to keep a sane outline.

P2

/ throttled · perf / UX
P2-11

Evidence

Under Slow 3G + 4× CPU throttling, the dashboard text eventually appears but wait_for for "TOTAL DOSSIERS" timed out at 60 s. Mid-load evaluate_script showed hasLoaderSpinner: false while widget cards rendered empty.

Add per-widget Skeleton components (project already has @/components/ui/skeleton). Render them while query.isLoading is true. Visible loaders dramatically improve perceived performance on slow connections without altering the unthrottled experience.
